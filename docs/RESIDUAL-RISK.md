# AERS Residual-Risk Register

Explicit, non-hidden residual risks after the production-assurance program. A
risk being listed here is the honest alternative to converting an incomplete
production integration into a `PASS`.

| ID | Risk | Why it remains | Mitigation / owner | Accepted until |
|----|------|----------------|--------------------|----------------|
| RR-01 | **No production verifier is deployed.** Production `VERIFIED` requires (a) a deployed verifier holding a private key, (b) a **pinned root** (`verifier._PINNED_ROOT_KEYS`, empty in this template) set via a control-plane change, and (c) an **audited signature backend**. Production trust is NOT granted by a file's filesystem location — production keys are accepted only from a trust bundle Ed25519-signed by a pinned root, which an authoring agent cannot forge. | Deploying/holding production keys and pinning a root is out of scope for a template. | `assess` reports `CTRL-EXTERNAL-VERIFIER` as `PARTIAL` (never `PASS`); `test_attacker_controlled_external_trust_file_is_refused` proves an agent-written external trust file is refused. Owner: security. | production rollout |
| RR-02 | **The bundled Ed25519 is a pure-Python reference; production requires an audited backend.** | Dependency-free offline operation; no audited crypto lib is present here. | The reference verifier is strict (canonical scalar `0<=S<L` — malleated `S+L` rejected — and canonical point encodings; RFC-8032 vector verified). But `production_valid` is gated on an **audited backend** (`cryptography`), which is absent here, so the reference impl can only ever return demo scope (`test_root_pinned_key_recognized_but_reference_backend_denies_production`). Production deployments install an audited library. Owner: security. | production rollout |
| RR-03 | **Private holdouts are contract-only.** The private-eval contract exists; real holdouts must live outside the author trust domain. | Holdouts inside a writable repo are not a boundary. | `CTRL-PRIVATE-ORACLE` reported `PARTIAL`. Owner: reviewer. | High Assurance adoption |
| RR-04 | **In-place default strengthenings are staged, not applied.** `require_second_reviewer_r2=true`, isolation-gate integration into `scripts/aers/verify.py`, and reviewer-independence enforcement live in `docs/proposed-control-plane-change/` with concrete patches and an integration test. | Applying them is a protected control-plane change that must not be made by the run proposing them (ADR-0004); the harness deny-list also blocks it. | Control-plane owner applies under the dogfooding workflow via a separate human PR. Until then, the additive isolation gate + benchmark enforce the behavior and the assurance claims say so explicitly. Owner: principal-architect. | next control-plane change window |
| RR-05 | **Evaluator-health covers the deterministic audit only.** Model-reviewer health needs live-model runs. | No live model is available offline. | Deterministic evaluator fully covered; add live-model canaries when a model is wired. Owner: reviewer. | live-model integration |
| RR-06 | **Isolation `PROVEN` depends on the host.** A host without a namespace mechanism cannot reach `PROVEN` locally. | Proof requires a real mechanism. | R2+ fails closed; run on trusted infra that supplies an attestation. Owner: sre. | infra provisioning |
| RR-07 | **Baseline comparison uses simulated trajectories.** The comparison harness distinguishes simulated from empirical evidence but has no live-model numbers yet. | No live model offline. | Clearly labeled simulated; re-run with a model for empirical numbers. Owner: reviewer. | live-model integration |
| RR-08 | **Two correlated reviewers can share a blind spot.** | Independence is structural, not perfect. | Evaluator-health monitors reviewer correlation; `THREAT-RUBBER-STAMP` tracks it. Owner: reviewer. | ongoing |
| RR-09 | **The assurance layer is not yet protected.** `scripts/aers_assure/**`, `scripts/assure.py`, and `assurance/**` are not in `protected-paths.json`/CODEOWNERS/the contract guardrail surfaces, so an autonomous task could in principle be scoped to modify the verifier/assessor and rely on it in the same run. | Adding them is a protected control-plane change that must not be made by the authoring run (ADR-0004); the harness deny-list also blocks it. | Staged patch `docs/proposed-control-plane-change/protect-assurance-layer.patch`; until applied, `assess` reports `CTRL-ASSURANCE-LAYER-PROTECTED` as FAIL (machine-visible). Owner: principal-architect/security. | next control-plane change window |
| RR-10 | **The author-side gate files and their wiring are not yet protected.** `scripts/checks/**` (both gates), `Makefile` (the `check:` wiring), and `assurance/reviews/config.json` (gate policy) are agent-writable and unowned; CI runs the branch's own gate code, so an agent could gut or un-wire a gate in the same PR it relies on. | Adding them to protected-paths/CODEOWNERS is a control-plane change the authoring run must not make (ADR-0004). | The gates were code-hardened to read policy and gated status from the committed baseline (self-relaxation in-PR has no effect) and to refuse empty tier lists; full protection is staged as `docs/proposed-control-plane-change/protect-gates.patch`. Owner: agent-platform/security. | next control-plane change window |
| RR-11 | **`validated_by` humanity is unverifiable by the gate.** The decision-log gate checks the handle is present and not the agent author id, but cannot prove a human set it. | A gate cannot authenticate a person. | CODEOWNERS on `.specify/specs/**` and `docs/decisions/**` (in `protect-gates.patch`) makes a real reviewer's approval the merge boundary; stated in `docs/decision-log-gate.md` limits. Owner: reviewer. | ongoing |
| RR-12 | **Self-declared decision risk and log completeness are not machine-checkable.** An agent can mislabel a one-way decision `cheap`, or simply not log a decision it made. | Risk labels and completeness are judgment, not facts a gate can derive. | The gate emits a computed salience line (entry + risky-entry counts) so reviewers can compare against the diff; reviewer skepticism protocol in `docs/decision-log-gate.md`. Owner: reviewer. | ongoing |
| RR-13 | **A pre-existing symlink pointing outside the repo can let a shell write escape both the pre-tool hook and the scope gate.** `pre_tool_guard` skips targets that resolve outside the repo. | Best-effort shell parsing is not the boundary; the scope gate covers in-repo protected paths and flags new escaping symlinks, but an already-committed out-of-repo symlink is a gap. | Run under the `agent_docs/sandbox-setup.md` container (the real egress/write boundary); reviewers reject unexplained symlinks. A `hooks.py` tightening can be added in a future control-plane change. Owner: agent-platform/sre. | sandbox hardening window |

## Handling

Every item above is time-bounded and owned. `PARTIAL`/`UNVERIFIABLE` in `assess`
and the residual-risk statements in `assurance/threats/threat-model.json` and the
assurance case are the machine-readable counterparts of this register.
